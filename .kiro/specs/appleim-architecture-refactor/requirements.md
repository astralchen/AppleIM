# AppleIM 架构优化需求规格

## 目标

将 AppleIM 从“页面用例和全能仓储聚合”收敛为更清晰的模块服务和账号 Store 架构，降低 Chat、Store、App 装配层的维护成本，并保持现有用户可见行为不变。

## 范围

- 重组 Store 层能力边界，避免上层继续依赖全能 `LocalChatRepository`。
- 拆分 Chat 业务接口，避免 `ChatViewModel` 只依赖一个过宽 `ChatUseCase`。
- 将会话列表、通讯录、搜索、登录收敛为模块级 Service 命名。
- 将账号生命周期、主界面依赖创建和本地数据删除流程从 `SceneDelegate` 下沉到 App 服务。
- 保持 UIKit、MVVM、Combine、async/await、Swift 6 严格并发和 iOS 15 兼容。

## 需求

### R1 Store 能力边界

作为维护 Store 层的工程师，我希望账号下的会话、消息、联系人、表情、通知设置、pending job、媒体索引和同步能力有清晰入口，以便修改某个能力时不需要理解整个 `LocalChatRepository`。

验收标准：

- 当上层模块需要数据能力时，应通过账号 Store 或窄协议获取，不直接把 `LocalChatRepository` 作为全能依赖传入。
- 当消息写入、未读清零、pending job、通知、索引或角标需要跨表更新时，应由 Store 组合层在统一事务或统一事件分发点处理。
- DAO 继续只负责表级或强相关表读写，不承担页面业务编排。
- 数据库 schema 可按未上架阶段更新当前基线，不要求生产迁移。

### R2 Chat 窄服务

作为维护聊天页的工程师，我希望聊天页按时间线、草稿、发送、消息操作、群聊、表情、模拟推送拆分依赖，以便测试和迭代单项能力时不需要实现完整聊天用例。

验收标准：

- `ChatViewModel` 不再以一个过宽 `ChatUseCase` 作为唯一依赖。
- 测试替身只需要实现当前测试关心的能力。
- 文本、图片、语音、视频、文件和表情发送仍保持原有状态流语义。
- 分页、最新消息观察、草稿恢复、@ 提及、群公告、语音播放状态、删除、撤回、重发和模拟收消息行为保持不变。

### R3 其他模块服务化

作为维护会话列表、通讯录、搜索和登录流程的工程师，我希望这些模块使用一致的 Service 命名和窄接口，以便分层约定统一。

验收标准：

- 会话列表、通讯录、搜索不再使用 `UseCase` 命名作为主要生产类型。
- ViewModel 仍只负责 UI 状态、用户输入、任务生命周期和状态发布。
- ViewModel 不直接访问数据库、Keychain、文件系统或底层网络 Client。
- 登录流程由登录会话服务负责编排认证和会话保存。

### R4 App 装配收敛

作为维护 App 启动和账号生命周期的工程师，我希望 `SceneDelegate` 只处理场景和窗口生命周期，以便账号状态和依赖装配可独立测试。

验收标准：

- `SceneDelegate` 不直接编排账号删除、本地数据清理、主依赖容器创建和服务端发送配置细节。
- `AppDependencyContainer` 负责模块服务工厂，不持有页面业务细节。
- 主界面构建仍使用 UIKit Tab/Nav，不引入 SwiftUI 重写。
- UI 测试配置继续强制使用隔离存储和 Mock/fail-first 发送服务。

### R5 并发和验证

作为项目维护者，我希望重构后仍满足 Swift 6 严格并发，以便不会引入新的并发警告或主线程越权访问。

验收标准：

- 新增协议和模型满足 `Sendable`。
- UI 类型和 ViewModel 保持 `@MainActor`。
- 数据库、媒体、搜索、同步、pending job 不在主线程执行。
- 完成后 `build-for-testing` 通过，相关单元测试和关键 UI 测试通过或明确记录外部环境阻塞。
