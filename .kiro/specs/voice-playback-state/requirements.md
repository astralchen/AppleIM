# 语音未播放红点需求规格

## 目标

将“消息已读”和“语音已播放”拆成两套状态，确保会话已读不会误消除收到语音消息的小红点。

## 范围

- 新增语音内容表播放状态字段。
- 更新 Store 模型、DAO、Repository 和聊天行状态映射。
- 保持文本、图片、视频、文件等非语音消息已读行为不变。
- 继续使用当前未上架阶段的数据库基线重建策略，不新增历史迁移链。

## 需求

### R1 独立播放状态

作为聊天用户，我希望收到的语音消息在没有播放过时始终显示小红点，即使这条消息已经被标记为已读。

验收标准：

- 收到的语音消息 `read_status == read` 且 `played_at == nil` 时，聊天气泡显示小红点。
- 收到的语音消息 `played_at != nil` 时，聊天气泡不显示小红点。
- 发出的语音消息不显示未播放小红点。

### R2 已读不等于已播放

作为聊天用户，我希望进入会话清除未读数时，不影响语音是否播放过。

验收标准：

- `markConversationRead` 仍将会话未读数归零。
- `markConversationRead` 仍将收到消息的 `read_status` 更新为已读。
- `markConversationRead` 不修改 `message_voice.played_at`。

### R3 播放才消除红点

作为聊天用户，我希望只有播放语音后，小红点才消失。

验收标准：

- `markVoicePlayed` 只更新语音内容表的 `played_at`。
- `markVoicePlayed` 不再修改消息主表的 `read_status`。
- 非语音消息调用 `markVoicePlayed` 时保持无副作用。

### R4 非语音行为保持不变

作为项目维护者，我希望本次修复不改变非语音消息和会话未读数逻辑。

验收标准：

- 文本、图片、视频、文件消息读取状态逻辑保持现状。
- 会话列表未读数和 App 角标统计保持现状。
- 现有发送、分页、观察最新消息流程保持现状。

### R5 数据库基线策略保持不变

作为项目维护者，我希望继续遵循未上架阶段的数据库基线策略。

验收标准：

- 提升 `DatabaseSchema.currentVersion`。
- 当前基线 schema 包含 `message_voice.played_at`。
- 不新增历史迁移脚本或迁移链。
